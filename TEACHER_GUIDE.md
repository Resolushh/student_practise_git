# Методические заметки для преподавателя

## Назначение шаблона

Этот проект можно опубликовать на GitHub и дать студентам ссылку для клонирования. Студенты должны продолжить работу локально в VS Code, выполнить Git-действия через графический интерфейс и оформить `README.md` как отчёт.

Задание адаптировано по практической работе `Git: README.md, .gitignore, ветки, merge, fetch, pull и конфликт`, но основной способ выполнения оставлен через интерфейс VS Code.

## Рекомендуемый сценарий проведения

1. Преподаватель публикует этот шаблон в своём GitHub-репозитории.
2. Студенты клонируют шаблон через VS Code.
3. Студенты продолжают работу локально в VS Code.
4. Каждый студент публикует результат в собственный GitHub-репозиторий.
5. Студент делает не менее 8 осмысленных коммитов, создаёт ветку, выполняет merge, проверяет Fetch/Pull, создаёт конфликт и оформляет отчёт.
6. На проверку сдаётся ссылка на репозиторий студента.

## Минимальный набор коммитов

Рекомендуйте студентам сделать не менее восьми коммитов:

1. `base: create git practice project`
2. `docs: add readme structure`
3. `chore: configure gitignore`
4. `docs: add git notes`
5. `docs: update readme with gitignore notes`
6. `docs: add project status`
7. `docs: update status locally`
8. `fix: resolve pull conflict in readme`
9. `docs: add practice screenshots to readme`
10. `docs: add final commit history screenshot`

## Проверяемые темы

Работа закрывает следующие темы:

- вход в GitHub через VS Code;
- клонирование репозитория;
- создание локального репозитория через `Initialize Repository`;
- добавление файлов в Stage;
- создание коммитов через поле `Message` и кнопку `Commit`;
- публикация проекта через `Publish Branch`;
- отправка изменений через `Push` / `Sync Changes`;
- получение изменений через `Pull`;
- проверка изменений через `Fetch`;
- различие между `Fetch` и `Pull`;
- Git Graph;
- создание и переключение веток;
- merge;
- базовое разрешение конфликтов;
- `.gitignore`;
- Markdown в `README.md`;
- добавление скриншотов в `README.md`.

## Требуемая структура

В финальном репозитории студента должны быть:

- `main.py`;
- `README.md`;
- `.gitignore`;
- `docs/notes.md`;
- `assets/screenshots/` с обязательными скриншотами.

Файлы `.env` и `temp.log` студент создаёт для проверки `.gitignore`, но они не должны попасть на GitHub.

## Проверка

При проверке удобно смотреть:

- наличие осмысленной истории коммитов;
- наличие отдельной ветки или следов merge в Git Graph;
- корректность `.gitignore`;
- наличие всех 10 обязательных скриншотов;
- совпадение текста отчёта с реальными действиями в репозитории;
- опубликован ли репозиторий на GitHub.

## Рекомендация по проверке Fetch/Pull

Самый понятный сценарий для студентов:

1. Добавить строку статуса в `README.md` локально и отправить на GitHub.
2. Изменить эту же строку прямо на GitHub и сделать commit.
3. Изменить эту же строку локально иначе и сделать локальный commit.
4. В VS Code выполнить `Fetch`.
5. Показать, что локальная и удалённая ветки разошлись.
6. Выполнить `Pull`.
7. Разрешить возникший конфликт в VS Code.

Так студент видит практическую разницу между командами, а не просто переписывает определение.

## Рекомендация по конфликту

В новой версии задания основной конфликт создаётся при `Pull` в `README.md`:

1. строка статуса изменяется на GitHub;
2. эта же строка изменяется локально в VS Code;
3. после `Pull` студент получает конфликт;
4. конфликт разрешается в редакторе VS Code.

Если на занятии мало времени, допустимо принять подробное описание конфликта, но максимальный балл лучше давать за реальную демонстрацию.

## Возможное усложнение

Для сильных студентов можно добавить одно из требований:

- дополнительно создать конфликт в `main.py` и реально разрешить его через VS Code;
- добавить в `main.py` консольное меню;
- оформить README с дополнительной таблицей коммитов;
- добавить ссылку на GitHub-репозиторий в отчёт;
- показать в скриншотах историю Git Graph до и после merge.
